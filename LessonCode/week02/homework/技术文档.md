# A.对于题目的敌人和子弹，内存对象的大小是固定的，所以采用数组形式的分配内存

## 0. 项目概述

本作业实现了一个基于 **空闲链表（free list）** 的 C++ 模板内存池 `MemoryPool<T>`，用于优化游戏中两类高频对象：

- **Bullet（子弹）**：频繁生成/销毁、数量多、生命周期短  
- **Enemy（敌人）**：频繁生成/销毁、数量相对少，但仍会在游戏循环中频繁发生

并对比两种方案的性能：

1) 普通 `new/delete`  
2) 使用内存池 `allocate/deallocate`

对比指标：
- 执行时间（`std::chrono`）
- 内存占用峰值（Windows 进程 Peak Working Set）

---

## 1. 内存碎片与池化技术理解

### 1.1 内存碎片
当程序大量、频繁地执行 `new/delete` 时，运行库（CRT）堆管理器会不断拆分、合并、管理小块内存。随着时间推移，堆中会出现各种大小的“空洞”，导致：

- **外部碎片**：空闲块分散，难以满足某些分配请求  
- **内部碎片**：对齐、块头（metadata）等原因造成“分配大于需求”的浪费

碎片的直接后果：
- 分配/释放耗时增加（查找、合并、锁竞争）
- 内存利用率降低
- 实时场景（游戏帧循环）出现延迟抖动（卡顿风险）

---

### 1.2 池化技术为什么能缓解碎片
**对象池/内存池**的核心思想是：  
> 预先申请一大块内存，将其切分为若干固定大小的块；之后“分配/回收”只在池内做指针操作，避免频繁触发系统堆分配器。

优势：
- 分配/回收基本为 O(1)（free list 头部弹出/压入）
- 运行时更稳定、延迟更可预测
- 只在初始化阶段做一次（或少量）系统分配，减少堆碎片产生

代价：
- 需要预分配容量（空间换时间）
- 池容量不足时需要失败返回或扩容策略（本作业为失败返回/保护）

---

## 2. 应用场景与设计思路（Bullet + Enemy）

### 2.1 Bullet 与 Enemy 
- 生成一些新子弹、新敌人
- 更新它们
- 一部分对象因碰撞/出界/死亡而被销毁

属于“短生命周期对象”如果使用传统 `new/delete` 下会出现：
- 分配器频繁工作
- 不稳定的性能开销
- 长时间运行碎片加剧

因此选择为 **Bullet 和 Enemy 各建立一个 MemoryPool**：
- MemoryPool<Bullet> bulletPool(...)
- MemoryPool<Enemy> enemyPool(...)

---

### 2.2 作业模拟的测试场景
使用 `SimConfig` 控制“不断生成和销毁”：

- 循环 `frames` 帧
- 每帧生成 `spawnBullets` 个子弹、`spawnEnemies` 个敌人
- 每帧销毁 `destroyBullets` 个子弹、`destroyEnemies` 个敌人
- 用 `std::vector<T*>` 保存“当前在场对象”
- 销毁策略：从 vector 尾部弹出（稳定且易实现）

示例配置（代码中使用）：
- `frames = 100`
- `spawnBullets = 100` / `destroyBullets = 100` → 子弹总创建次数约 10000
- `spawnEnemies = 10` / `destroyEnemies = 10` → 敌人总创建次数约 1000

该设计符合“不断生成与销毁”的游戏循环特点。

## 3. 实现细节与关键代码解析

### 3.1 内存块 Chunk：union 复用内存
内存池使用 union 复用同一段内存：

- 空闲状态：存储 `next` 指针（把所有空闲块串成链表）
- 使用状态：存储 `T` 对象所需的原始字节（placement new 构造）

关键结构：
```cpp
union Chunk {
    Chunk* next;
    alignas(T) unsigned char data[sizeof(T)];
};
```

关键点说明：
- `alignas(T)`：保证 `data` 满足 `T` 的对齐要求，避免未对齐访问导致未定义行为
- `sizeof(T)`：保证每块足以容纳一个 `T`

---

### 3.2 初始化：一次性分配连续数组并建立 free list
构造函数核心逻辑：
```cpp
m_pool = new Chunk[m_allCapacity];
m_freeList = m_pool;

for (std::size_t i = 0; i + 1 < m_allCapacity; ++i) {
    m_pool[i].next = &m_pool[i + 1];
}
m_pool[m_allCapacity - 1].next = nullptr;
```

这样每个空闲块通过 `next` 串起来：
- `allocate()` 从 `m_freeList` 取头节点
- `deallocate()` 把块头插回 `m_freeList`

连续内存的好处：
- 初始化简单
- 更好的缓存局部性（CPU cache locality）
- 析构时一次 `delete[] m_pool` 即可释放全部资源

---

### 3.3 分配：placement new 在已有内存上构造对象
`allocate()`：
```cpp
Chunk* chunk = m_freeList;
m_freeList = m_freeList->next;
return new (chunk->data) T();
```

解释：
- `new (chunk->data) T()` 是定位 new（placement new）
- 不向系统申请内存，只在 `chunk->data` 指向的原始内存中“原地构造”一个 T 对象并返回 `T*`

---

### 3.4 回收：显式析构 + 归还空闲链表
`deallocate()`：
```cpp
ptr->~T();
Chunk* chunk = reinterpret_cast<Chunk*>(ptr);
chunk->next = m_freeList;
m_freeList = chunk;
```

解释：
- 由于对象是 placement new 构造，不能用 `delete ptr`
- 必须显式调用析构函数 `~T()`，释放对象内部资源（若有）
- 再把内存块作为 `Chunk` 节点归还给 free list

---

### 3.5 禁用拷贝：避免 double free
```cpp
MemoryPool(const MemoryPool&) = delete;
MemoryPool& operator=(const MemoryPool&) = delete;
```

原因：
- 内存池内部持有 `m_pool` 指针（指向 `new[]` 得到的内存）
- 默认浅拷贝会导致两个池对象指向同一块内存
- 析构时会 `delete[]` 两次，产生未定义行为（崩溃风险）

## 4. Benchmark 设计与指标

### 4.1 时间测量
使用：
- `std::chrono::high_resolution_clock::now()`
- `duration_cast<std::chrono::microseconds>`

分别统计：
- `benchmarkNormal(cfg)`：`new/delete` 版本
- `benchmarkPool(cfg)`：两个内存池版本

---

### 4.2 内存峰值测量
使用 Windows API 获取进程内存信息：
```cpp
GetProcessMemoryInfo(GetCurrentProcess(), &pmc, sizeof(pmc));
return pmc.PeakWorkingSetSize;
```

- PeakWorkingSetSize 表示进程“历史峰值工作集”（字节数）
- 该值只升不降，因此在同一进程内重复跑多轮 benchmark 时，第二轮峰值可能被第一轮抬高并保持不变
- 若要严格对比峰值，建议分两次启动程序分别测 Normal 与 Pool（本作业仍保留 repeat 作为稳定性观察）

## 6. 优化前后对比分析

### 6.1 执行时间

![.\res\9.png](.\res\9.png)

**优化前（new/delete）**
- 每次分配/释放会进入堆管理器
- 有额外元数据管理、锁竞争、碎片处理等开销
- 在高频对象生成/销毁场景下耗时明显且可能抖动

**优化后（MemoryPool）**
- 分配/回收主要是指针操作（free list）
- placement new/显式析构只处理对象构造/析构，不进行系统级内存申请/释放
- 时间通常显著下降，性能更稳定

---

### 6.2 内存占用峰值

![.\res\10.png](.\res\10.png)

**优化前（new/delete）**

- 内存由堆管理器动态增长，峰值取决于堆策略与碎片情况

**优化后（MemoryPool）**

- 初始化时预分配内存，峰值通常会更高一些（空间换时间）
- 优点是运行时分配更稳定，减少碎片风险
